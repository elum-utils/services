# Контракт Импорта И Экспорта

## Items

`reference` — единственный источник истины для item и его данных: типа,
редкости, позиции, названия и описания.

Остальные сервисы хранят только ключ item в собственной связи с наградой или
продуктом, например `reward.key` или `item_id`.

- Export пакеты сервисов, кроме `reference`, не содержат верхнеуровневый
  раздел `items`.
- Import сервисов, кроме `reference`, не создает, не обновляет, не валидирует
  и не считает items.
- Нет `dependencies.items` и нет требования сначала импортировать Reference
  пакет.
- Ключ награды принимается как непрозрачная ссылка. Когда приложению нужно
  показать item пользователю, оно получает его представление из Reference.

Правило исключает дублирование метаданных и расхождение разных копий одного
item между сервисами.

## Переход

`reference` сохраняет собственный export/import items, потому что является их
владельцем. Остальные существующие сервисы приводятся к этому контракту
отдельными изменениями: сначала удаляется верхнеуровневый export/import
`items`, затем при необходимости переносится их устаревшая локальная модель
на Reference. Такой переход не должен механически удалять данные из рабочих
таблиц.

## Bulk Import

Каждый сервис сам определяет JSON-структуру, бизнес-валидацию, порядок записи
связанных таблиц и SQL-конфликт для своего import. Общей является только
механика разбиения bulk SQL на батчи из
`internal/utils/importexport`.

- `ForEachBatch` ограничивает команду числом строк и числом параметров.
- Стандартный лимит: до 1000 строк и до 60 000 параметров на команду; он
  оставляет запас до ограничения PostgreSQL в 65 535 параметров.
- Repository конкретного сервиса передаёт число SQL-колонок и строит только
  статический SQL для своих таблиц. Имена таблиц, список колонок и
  `ON CONFLICT` не принимаются из import JSON.

При миграции сервиса на этот контракт нужно добавить тест пакета, который
превышает предел параметров для его самой широкой bulk-вставки.

## Consistency

- Export читает связанные таблицы в одной `REPEATABLE READ READ ONLY`
  транзакции. Файл не должен смешивать состояние до и после параллельного
  admin update.
- Import валидирует весь пакет до открытия write-транзакции: корневые сущности,
  localization, rewards, target, дубли и числовые границы PostgreSQL.
- Preview конфликтов и bulk write выполняются в одной транзакции под
  transaction-level advisory lock конкретной workspace.
- Обычные admin catalog writes используют тот же lock. Иначе
  `fail_on_conflict` и `skip_existing` не имеют корректной семантики при
  конкурентном запросе.
- Cache version изменяется только после успешного commit. Ошибка внешнего cache
  backend передается в диагностический callback и не превращает успешный DB
  write в ошибку API.

Минимальные integration tests для каждого import/export: invalid preflight без
частичной записи, все conflict strategies, пакет больше лимита параметров,
конкурентный admin write и согласованный export snapshot.
